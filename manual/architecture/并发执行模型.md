# 并发执行模型

本文档描述 leapvm_server 的 WorkerPool 并发架构。

## 1. 架构

```text
leapvm_server
  └─ WorkerPool
       ├─ Worker 0 ─── Thread ─── VmInstance (V8 Isolate + Context)
       ├─ Worker 1 ─── Thread ─── VmInstance
       ├─ Worker 2 ─── Thread ─── VmInstance
       └─ Worker N ─── Thread ─── VmInstance
```

每个 Worker 线程持有独立的 V8 Isolate，完全隔离。Worker 在自己的线程上创建和销毁 VmInstance，无跨线程析构问题。

## 2. 任务分派

- **Lock-free round-robin**：原子计数器分配，最小化争用
- 每个 Worker 有独立的任务队列（mutex 保护）
- IpcServer 收到 `run_signature` 请求后调用 `WorkerPool::SubmitTask()`

### 异步回调模式

1. `SubmitTask(TaskRequest, callback)` 将任务放入目标 Worker 队列
2. Worker 线程执行完毕后调用 callback，将结果推入 completion queue + 写 eventfd
3. IpcServer 主循环 `Poll()` 排空 completion queue，发送响应

## 3. Worker 生命周期

### 初始化

1. 创建 `VmInstance`（独立 V8 Isolate + Context）
2. 配置 Hook 黑名单
3. 执行 bootstrap 脚本（注入 `__runtimeBootstrap`）
4. 执行 wrapped bundle（带 code cache）
5. 状态 → `kIdle`

### 任务执行

1. 从队列取出 TaskRequest
2. 状态 → `kBusy`
3. 解析缺省字段（未携带的 snapshot → 预加载默认值）
4. 执行 CombinedTask 或 SplitCachedTask
5. 状态 → `kIdle`
6. 计数 `tasks_completed++`

### Recycle

当 Worker 执行任务数达到 `maxTasksPerWorker`（默认 200）：

1. 状态 → `kRecycling`
2. 销毁旧 VmInstance
3. 创建新 VmInstance（同一线程）
4. 重新执行 bootstrap + bundle（使用共享 code cache）
5. 状态 → `kIdle`

Recycle 目的是防止长期运行的内存膨胀。共享 code cache 不受 recycle 影响。

## 4. Code Cache 共享

- Bundle code cache 和 Target code cache 在 `WorkerPool::Start()` 生成一次
- 所有 Worker 以只读方式共享
- Worker recycle 后继续使用同一份 cache
- 详见 `Code-Cache预编译缓存.md`

## 5. 统计

通过 `get_stats` IPC 请求可查询：

| 字段 | 含义 |
|------|------|
| `total_workers` | Worker 总数 |
| `idle_workers` | 空闲 Worker 数 |
| `busy_workers` | 忙碌 Worker 数 |
| `recycling_workers` | 正在 recycle 的 Worker 数 |
| `total_tasks_completed` | 累计完成任务数 |
| `total_tasks_failed` | 累计失败任务数 |
| `pending_tasks` | 等待队列中的任务数 |
| `target_cache_hits` | 累计 target cache 命中 |
| `target_cache_misses` | 累计 target cache 未命中 |
| `target_cache_rejected` | 累计 target cache 被 V8 reject |

## 6. V8 Isolate 隔离红线

- 每个 Worker 线程持有独立 Isolate，不可跨线程共享 V8 对象
- 禁止 `static v8::Eternal<v8::Private>` 跨 Isolate 共享 brand key
- 必须用 `v8::Private::ForApi()` 逐 Isolate 获取
- ICU 符号带 `_leapvm` 后缀与 Node.js 隔离

## 7. 历史说明

Node.js 侧的 ThreadPool / ProcessPool（`leap-env/src/pool/`）已废弃，不再进入 bundle。并发能力完全由 leapvm_server 的 C++ WorkerPool 管理。
