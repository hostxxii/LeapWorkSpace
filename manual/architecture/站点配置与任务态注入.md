# 站点配置与任务态注入

本文档描述 `siteProfile` 的结构、校验、合并、任务态注入链路和清理边界。

## 1. 职责

本模块承担两项工作：

1. 解析并校验站点基线配置 `siteProfile`。
2. 在每次任务前后写入和清理 `signatureTaskState` 及其派生状态。

它不负责：

- 目标脚本预加载
- code cache
- worker 常驻缓存
- bundle 装载

这些由 `runner.js`、`TaskProtocol`、`WorkerPool` 负责。

## 2. `siteProfile` 结构

```json
{
  "id": "jd-search-pc",
  "overrideMode": "strict",
  "requiredFields": [
    "fingerprintSnapshot.navigator.userAgent",
    "fingerprintSnapshot.navigator.languages",
    "fingerprintSnapshot.location.href"
  ],
  "fingerprintSnapshot": {
    "navigator": {
      "userAgent": "...",
      "platform": "Win32",
      "languages": ["zh-CN"],
      "plugins": [],
      "mimeTypes": [],
      "permissions": {}
    },
    "screen": { "width": 1920, "height": 1080 },
    "windowMetrics": { "innerWidth": 1920, "innerHeight": 969 },
    "location": { "href": "https://www.jd.com", "origin": "https://www.jd.com" },
    "performanceSeed": {},
    "featureFlags": {},
    "randomSeed": "seed",
    "canvasProfile": {}
  },
  "storagePolicy": { "localStorage": "replace", "sessionStorage": "replace" },
  "storageSnapshot": { "localStorage": {}, "sessionStorage": {} },
  "documentSnapshot": { "cookie": "...", "referrer": "...", "lastModified": "..." }
}
```

## 3. 校验与模式

`validateSiteProfile()` 在 `runner.js` 中执行，规则如下：

- `siteProfile` 必须是 plain object。
- `overrideMode` 只接受 `merge` 或 `strict`。
- `strict` 模式下，`requiredFields` 必须是非空字符串数组，且每条路径必须已存在。
- `fingerprintSnapshot.navigator.languages` 必须是 `string[]`。
- `storageSnapshot.localStorage` / `sessionStorage` 必须是对象或 `null`。
- `documentSnapshot.cookie` 必须是 `string` 或 `null`。
- `storagePolicy.localStorage` / `sessionStorage` 只接受 `replace` 或 `merge`。

## 4. 有效任务快照的合成

任务生效值 = `siteProfile` 基线 + per-task 显式覆盖。

`buildEffectiveTaskOverrides()` 使用 `deepMergeReplaceArrays()`：

- 对象递归合并
- 数组整体替换
- 显式任务字段优先

即：

```text
effectiveFingerprint = deepMerge(siteProfile.fingerprintSnapshot, task.fingerprintSnapshot)
effectiveStorage     = deepMerge(siteProfile.storageSnapshot,    task.storageSnapshot)
effectiveDocument    = deepMerge(siteProfile.documentSnapshot,   task.documentSnapshot)
effectivePolicy      = deepMerge(siteProfile.storagePolicy,      task.storagePolicy)
```

## 5. 服务端注入链路

服务端通过 `TaskProtocol` 生成脚本：

- Combined 路径：`BuildCombinedScript()`
- Split Cached 路径：`BuildTaskSetupScript()` + target + `BuildTaskCleanupScript()`

setup 阶段的核心调用顺序是：

```javascript
leapenv.beginTask(taskId);
domService.beginTaskScope(taskId);
leapenv.applyFingerprintSnapshot(fp);
leapenv.applyStorageSnapshot(storage, policy);
leapenv.applyDocumentSnapshot(doc);
```

cleanup 阶段的核心调用顺序是：

```javascript
leapenv.resetSignatureTaskState();
domService.endTaskScope(taskId);
leapenv.endTask(taskId);
```

## 6. `applyFingerprintSnapshot()`

### 6.1 写入的任务态字段

`signature-task.instance.js` 会重置并写入：

- `navigator`
- `screen`
- `windowMetrics`
- `performanceSeed`
- `featureFlags`
- `randomSeed`
- `canvasProfile`

其中 `navigator` 还会特别处理：

- `languages`
- `plugins`
- `mimeTypes`
- `permissions`

这些字段直接影响 `Navigator`、`PluginArray`、`MimeTypeArray`、`PermissionStatus`、`Window`、`Crypto`、`Canvas/WebGL` 等 impl。

### 6.2 伴随副作用

`applyFingerprintSnapshot()` 不只是写一个 JSON：

1. 更新 `location` 快照。
2. 在存在 `performanceSeed` 时调用 `performanceImplHelpers.reset()`。
3. 确保 document 默认树已建立。
4. 为 `canvasProfile`、`randomSeed`、插件集合等高指纹面提供运行时输入。

## 7. `applyStorageSnapshot()`

按 `storagePolicy` 分别处理 `localStorage` / `sessionStorage`：

- `replace`：先 `clear()` 再写入。
- `merge`：保留已有 key，snapshot 覆盖同名项。

storage 数据不是直接挂在 `signatureTaskState`，而是写入实际 storage 实例。

## 8. `applyDocumentSnapshot()`

当前覆盖：

- `cookie`
- `referrer`
- `lastModified`

实现会先确保默认 document 树存在，再把字段写入 document 私有状态。

## 9. `resetSignatureTaskState()`

### 9.1 清理的状态面

任务结束后必须清理：

- `navigator`
- `screen`
- `windowMetrics`
- `performanceSeed`
- `featureFlags`
- `randomSeed`
- `canvasProfile`
- `localStorage`
- `sessionStorage`
- `location` / `history`
- `document.cookie`
- `document.referrer`
- `document.lastModified`

### 9.2 额外 reset 钩子

除通用字段外，还会额外尝试调用：

- `resetWindowTaskState()`
- `resetMessagePortTaskState()`

原因：

- `WindowImpl` 内部有任务级缓存和派生状态。
- `MessagePort` / `MessageChannel` 维护端口队列和 live 对象，必须在任务后回收。

如果不做这两步，跨任务泄漏会落在“看不见的运行时状态”，而不是简单 JSON 字段上。

## 10. 与其他模块的边界

- `Window与全局对象模型.md`
  读取 `windowMetrics`、`randomSeed`，并暴露 BOM 单例。
- `特殊API模块.md`
  读取 `canvasProfile`、`plugins`、`mimeTypes`、`permissions`、消息端口状态。
- `运行时入口与任务执行链路.md`
  负责解释调用顺序和 `try/finally` 包裹。
- `V8 Platform与任务调度.md`
  负责解释这些脚本如何被投递到 VM thread 执行。

## 11. 文件来源

站点配置文件存放在 `site-profiles/`，可通过两种方式进入系统：

- `initializeEnvironment({ standalone: { siteProfilePath } })`
- `leapvm_server --site-profile PATH`

服务端启动时只提取 snapshot 字段做预加载回退，不把整个 `siteProfile` 原样常驻到 Worker。
