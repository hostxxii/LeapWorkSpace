# 运行时入口与任务执行链路

本文档描述 Node.js 侧的运行时入口（`runner.js`）和 VM 内 bundle 执行时的 bootstrap / 任务 setup / cleanup 全链路。

## 1. runner.js 三主函数

`leap-env/runner.js` 对外暴露三个核心异步函数：

### initializeEnvironment(options)

1. 解析配置：`resolveRunOptions()` 合并默认值
2. 准备预加载目标脚本：写入临时文件或使用显式路径
3. 创建 `ServerManager` 并 `start()`：启动 leapvm_server 子进程
4. 创建 `StandaloneClient` 并 `connect()`：建立 TCP 连接
5. 返回 `{ leapvm, resolved, inspectorInfo }`

`leapvm` 持有 `_client`、`_serverManager`、临时文件路径等句柄。

### executeSignatureTask(leapvm, task)

1. 校验 `siteProfile`（strict/merge 模式）
2. 深合并 siteProfile + per-task overrides → `buildEffectiveTaskOverrides()`
3. 构建 payload：`{ beforeRunScript, resourceName, fingerprintSnapshot, storageSnapshot, documentSnapshot, storagePolicy }`
4. 调用 `client.runSignature(payload)` 发送 TCP 请求
5. 返回 `response.result`

### shutdownEnvironment(leapvm)

1. `client.shutdown()` → 请求服务端优雅停机
2. `client.disconnect()` → 关闭 TCP
3. `serverManager.stop()` → SIGTERM + 超时 SIGKILL
4. 清理临时目标脚本文件

## 2. siteProfile 合并逻辑

任务快照的有效值 = siteProfile 基线 + per-task 显式覆盖。

合并规则（`deepMergeReplaceArrays`）：
- 对象递归合并
- 数组整体替换（不做元素级合并）
- 显式传入的 per-task 字段优先

严格模式（`overrideMode: 'strict'`）下，`requiredFields` 中列出的路径必须在 siteProfile 中存在，否则直接抛错。

## 3. VM 内 bootstrap 注入

服务端在 bundle 执行之前，通过 `TaskProtocol::BuildBootstrapScript()` 注入：

```javascript
globalThis.leapenv = globalThis.leapenv || {};
globalThis.leapenv.__runtimeBootstrap = {
  domBackend: 'dod',
  bridgeExposureMode: 'strict',
  globalFacadeMode: 'strict',
  enableInspector: false,
  inspectorPort: 9229,
  targetVersion: '...'
};
```

这是 bundle 内 `config.js` 读取配置的来源。

## 4. bundle 执行顺序

bundle 由 `generate-entry.js` 按以下分组和顺序打包：

```
core/         runtime.js → config.js → tools.js → skeleton-loader.js
skeleton/type/    按继承拓扑排序（Kahn 算法，父类先于子类）
skeleton/instance/  window.instance.skeleton.js 强制排首
impl/             按文件名字典序
instance/         按文件名字典序
```

bundle 格式为 IIFE（esbuild `format: 'iife'`），在 VM 内直接执行。

## 5. bundle 内初始化链

bundle 执行后的内部状态：

1. **runtime.js**：建立 `leapenv.__runtime`（config、host timers、debug、task、bridge、perf）
2. **config.js**：从 `__runtimeBootstrap` 读取配置，写入 `leapenv.config`
3. **tools.js**：注册工具函数
4. **skeleton-loader.js**：调用 `nativeBridge.defineEnvironmentSkeleton(envDescriptor)` 把全部 skeleton 数据传入 C++ bridge
5. **skeleton 文件**：注册各 type/instance skeleton
6. **impl 文件**：`registerImpl()` 注册行为实现，预缓存原型链描述符

## 6. 任务执行链路（VM 内）

服务端通过 `TaskProtocol::BuildCombinedScript()` 生成单个脚本块：

```javascript
{
  // 块作用域捕获运行时引用
  const __leapEnv = globalThis.leapenv;
  const __leapDomService = __leapEnv?.__domService;
  const __leapRuntime = __leapEnv?.__runtime;
  const __leapHookRuntime = __leapRuntime?.debug?.hookRuntime;

  try {
    // ── setup ──
    __leapHookRuntime.active = false;
    __leapHookRuntime.phase = 'setup';
    __leapEnv.beginTask(taskId);
    __leapDomService.beginTaskScope(taskId);
    __leapEnv.applyFingerprintSnapshot(fp);
    __leapEnv.applyStorageSnapshot(storage, policy);
    __leapEnv.applyDocumentSnapshot(doc);

    // ── transition ──
    __leapHookRuntime.phase = 'task';
    __leapHookRuntime.active = true;

    // ── target ──
    // ... 目标签名脚本 ...

  } finally {
    // ── cleanup ──
    __leapHookRuntime.active = false;
    __leapHookRuntime.phase = 'idle';
    __leapEnv.resetSignatureTaskState();
    __leapDomService.endTaskScope(taskId);
    __leapEnv.endTask(taskId);
  }
}
```

### 设计要点

- **块作用域捕获**：在块顶部捕获 `__leapEnv` 等引用，即使目标脚本篡改 `globalThis.leapenv` 也不影响 cleanup
- **try/finally 保证清理**：目标脚本抛错时仍然执行 resetSignatureTaskState
- **Hook 阶段切换**：setup 阶段不记录 Hook 日志，task 阶段正常记录，cleanup 阶段关闭记录

### resetSignatureTaskState 必须清理的字段

- navigator / screen 快照覆盖
- localStorage / sessionStorage
- document.cookie / document.referrer
- randomSeed
- layoutRect 缓存

遗漏任何字段会导致跨任务指纹泄漏。

## 7. 配置优先级

```
__runtimeBootstrap.xxx  >  环境变量 LEAP_XXX  >  硬编码默认值
```

当前硬编码默认值：

| 配置 | 默认值 |
|------|--------|
| `dispatchMissingMode` | `'warn'` |
| `domBackend` | `'dod'` |
| `bridgeExposureMode` | `'strict'` |
| `globalFacadeMode` | `'strict'` |
