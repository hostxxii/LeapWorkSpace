## 问题与改造摘要

### 1. Hook 拦截内置对象导致环境代码异常

历史问题：

- `NamedPropertyHandler` 会拦截 `window` 上的内置对象访问（如 `Map`、`Object`、`Symbol`）
- 补环境代码在 VM 内执行时，可能因 Hook 行为导致异常或噪音日志

历史方案（来源记录）：

- 通过 `setPropertyBlacklist()` 将内置对象、噪音属性、内部前缀加入黑名单
- 在运行环境代码前先配置黑名单，再执行 `runScript`

迁移说明（当前手册口径）：

- Hook/黑白名单机制仍在 LeapVM 主线中，但实现已演进，当前应以 `manual/reference/LEAPVM_API.md` 与相关架构文档为准
- 当前 Hook 过滤顺序以代码实现为准：白名单 gate → 黑名单过滤（已在根手册中校验）

### 2. 自动化构建系统：`entry.js` 自动生成

历史问题：

- 早期需要手工维护 `entry.js` import 列表
- 容易遗漏文件、顺序错误，维护成本高

落地方案（来源记录）：

- 引入 `generate-entry.js` 自动扫描 `src/` 模块文件并按规则排序
- 配合数字前缀命名约定，保证加载顺序稳定
- `npm run build` 统一执行“生成入口 + 打包”

沉淀价值：

- 将“加载顺序”从人工维护迁移到规则驱动（命名 + 目录优先级）
- 为后续 skeleton / impl 扩展提供低摩擦接入方式

### 3. 构建目录集中化与目录结构整理

历史问题：

- 构建脚本、生成脚本、输出目录曾分散在项目根目录
- 根目录噪音较大，构建逻辑与业务逻辑混杂

落地方案（来源记录）：

- 将构建相关内容集中到 `src/build/`
- 根目录保留 `index.js` / `index_debug.js` / `package.json` 等入口文件
- 配合自动化构建，形成更清晰的项目结构

当前状态（2026-02-22）：

- `leap-env/package.json` 中仍由 `src/build/build.js` 作为构建入口（已在命令手册收敛）
- 根目录统一回归入口已迁至仓库根 `tests/`（PowerShell runners）

### 4. 架构演进（历史语境）

来源记录梳理了三阶段演进：

- 旧架构（已废弃）：JS 侧直接尝试创建/模拟 `window`
- 三层架构（历史）：`def/impl/instance` 分层但仍需手动维护入口
- v2 自动化构建阶段（现行方向）：`core` + `skeleton` + `impl` + `instance` + `build`
