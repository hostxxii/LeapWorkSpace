# 2026-02-23_测试脚本与压测脚本物理迁移评估

日期：2026-02-23  
来源：`REPO_REORG_TODO.md` J1（后续可选评估项）

## 结论（先给结论）

建议采用“**逐步物理迁移 + 保留兼容壳 + 以新增脚本优先落根**”策略，而不是一次性将现有 `leap-env/`、`leap-vm/` 下所有测试/压测脚本整体迁到根 `tests/`。

换句话说：

- **要迁**（方向正确）：根 `tests/` 作为最终统一入口与长期归档位置
- **不一次性硬迁**（降低风险）：先保证回归稳定与路径兼容

## 评估背景

当前仓库已经完成：

- 根目录统一编排（`tests/runners/*` + `tests/manifest/*`）
- 回归结果归档（`tests/results/`）
- 基线与门限文档（`tests/baselines/`）

当前未完成的是：

- 将历史散落在 `leap-env/scripts/`、`leap-vm/*.js` 的脚本**物理搬迁**到根 `tests/` 目录（而非仅通过根入口调用）

## 风险与收益评估

### 主要收益

1. **统一发现与导航**
   - 新同事只看 `tests/` 即可定位回归/压测入口
   - 脚本命名、输出规范、结果归档更一致

2. **降低跨目录耦合感知成本**
   - 目前命令来自多个模块目录，阅读 manifest 仍需跳转多处
   - 迁移后可逐步把测试数据、夹具、说明聚拢到根目录

3. **利于后续扩展 JSON 结构化摘要 / CI 接入**
   - 脚本路径统一后，元数据和报告聚合更容易

### 主要风险

1. **路径依赖与相对路径破坏**
   - 现有脚本可能依赖 `__dirname`、相对资源路径、默认工作目录
   - 物理搬迁会触发隐蔽路径问题（尤其是读取 fixture / bundle / build 输出）

2. **模块边界语义变弱**
   - 某些脚本更接近模块内部调试脚本，而非根级回归资产
   - 全部迁到根 `tests/` 可能让模块维护者更难定位“本模块自测入口”

3. **回归噪音与历史路径失效**
   - 短期内会导致 README/文档/个人脚本/命令历史中的路径大量失效
   - 需要兼容壳或映射层，否则迁移收益被维护成本抵消

## 建议策略（分阶段）

### Phase J1-A（已具备）

- 保持“根编排 + 子模块脚本物理保留”的现状作为稳定基线
- 新增功能优先接入根 `tests/manifest/*` 与 `tests/runners/*`

### Phase J1-B（建议下一步）

对“跨模块集成测试 / 压测脚本”先迁，不先迁模块内单元/调试脚本。

优先候选（示意）：

- 跨模块集成（同时依赖 `leap-env` + `leap-vm`）
- 长期保留的 perf 基线脚本（已进入 `tests/manifest/perf.ps1`）
- 需要统一产物归档的脚本

暂缓候选（示意）：

- 临时调试脚本
- 强依赖模块内部相对路径的实验脚本
- 尚未纳入根 manifest 的一次性验证脚本

### Phase J1-C（物理迁移实施规则）

建议采用“**复制迁移 + 兼容壳**”而非直接剪切：

1. 在根 `tests/scripts/` 新建目标脚本（按功能分目录）
2. 原路径保留薄包装（wrapper）并打印迁移提示（或转发调用）
3. 根 `manifest` 先切到新路径
4. 连续若干轮回归稳定后，再删除旧脚本

推荐目录草案：

- `tests/scripts/integration/`
- `tests/scripts/perf/`
- `tests/scripts/manual/`
- `tests/fixtures/`（按需）

## 迁移准入标准（建议）

脚本满足以下条件再迁移：

- 已纳入 `tests/manifest/*`（有统一编排价值）
- 相对路径依赖已清点并可改为 repo-root 解析
- 输出目录可落到 `tests/results/` 或 runner 已接管输出
- 至少经过一轮 `smoke` / `full` / `perf` 中对应档位验证

## 对当前仓库的建议执行结论（本次评估）

- **J1 结论：通过（可执行）**
- **本轮不做大规模物理迁移实现**（避免引入路径回归）
- 后续若继续推进，建议先选 1-2 个 `perf` 脚本做样板迁移并保留兼容壳
