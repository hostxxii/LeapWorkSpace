## 背景

来源记录描述了 LeapVM 的核心技术前提：在 Node.js 进程中运行一个与 Node 自带 V8 完全隔离的 V8 实例（用于 LeapVM Native Addon）。

历史崩溃现象（来源记录）：

- `Assertion failed: (data.first) != nullptr at node::NodePlatform::ForIsolate`

来源记录给出的根因判断是：代码在链接/运行时走到了 Node.js 的 V8 符号，而不是 addon 内部静态链接的 `v8_monolith.lib`，从而触发 Node 平台上的 Isolate 注册断言。

## 问题

来源记录集中解决的是“双 V8 共存”相关的几个关键误区与实现难点：

### 1. 符号与全局状态误绑定到 Node.js V8

- 在 Node.js 进程内创建 V8 Isolate 时，如果最终调用的是 Node 内部 V8 的符号，就会被 Node 的 Platform 管理逻辑约束
- 这会导致“未在 Node Platform 上注册”的断言或运行时异常

### 2. 常见误区：认为一个进程只能有一个 V8

来源记录明确指出这类结论通常隐含了一个前提：

- 所有代码都共享同一个 V8 库（符号与全局状态不隔离）

而 LeapVM 的方案是在 addon 内静态链接自己的 V8，避免共享 Node 的 V8 实现与全局状态。

### 3. 构建与链接链路极易出错

来源记录中强调以下配置若处理不当，会导致“看似编译通过但运行崩溃”：

- V8 头文件优先级
- `v8_monolith.lib` 链接顺序
- `/WHOLEARCHIVE` 保活符号
- `/FORCE:MULTIPLE` 处理重复符号
- CRT 版本一致性（如 `/MT` vs `/MTd`）

## 方案（来源记录摘要）

### 1. 静态链接的双 V8 共存

来源记录的核心洞察：

- `node.exe` 使用其自身 V8（嵌入在 Node 进程环境中）
- `leapvm.node` 静态链接 `v8_monolith.lib`，拥有自己的一套 V8 符号与全局状态
- Windows 加载器会为 DLL / `.node` 维护独立模块数据段，因此 addon 内 V8 全局状态与 Node 的 V8 不共享

结果：

- `leapvm.node` 内部的 `v8::V8::platform_`、`v8::Isolate` 等与 Node 的对应对象相互隔离
- 可以在同一进程内维持两套独立 V8 生命周期

### 2. 构建链路关键点（CMake / MSVC）

来源记录反复强调的关键配置（摘要）：

- 宏定义与 V8 编译配置一致（如压缩指针、sandbox、静态链接）
- V8 头文件优先于 Node-API 头文件参与编译解析
- `v8_monolith.lib` 在 `target_link_libraries` 中置前，保证符号优先解析到 addon 自身 V8
- MSVC 使用 `/WHOLEARCHIVE` 保留 V8 符号
- 使用 `/FORCE:MULTIPLE` 处理与 Node 侧重复符号
- Release 配置与 V8 所用 CRT 保持一致（来源记录中强调 `/MT`）

### 3. 必须初始化“自己的” V8 Platform

来源记录中专门纠正的错误做法：

- 不能因为“Node 已初始化 V8”就跳过 addon 内部 V8 的 `InitializePlatform/Initialize`
- LeapVM 必须初始化 `leapvm.node` 自己静态链接的 V8 Platform

这对应来源记录中 `v8_platform.cc` 的初始化逻辑说明。

### 4. 避免错误方案

来源记录明确列出两类误区：

- 跳过 Platform 初始化（会导致后续创建 Isolate 崩溃）
- 直接复用 Node 的内部 Platform API（绑定 Node 内部实现、丢失版本独立性）

## 验证（来源记录摘要）
