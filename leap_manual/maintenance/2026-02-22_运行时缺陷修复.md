## 背景

在 A3 iframe、多上下文、定时器与 Inspector 场景推进后，出现一组已知运行时缺陷，需要先做风险分级，再按优先级落地修复。

原始记录在 `SOLUTION.md` 中分为两段：

- 已知缺陷分析与修复计划（BUG_FIX_PLAN）

## 问题（按优先级）

### P0：`child_frames_` 无销毁接口 → OOM

- `ChildFrame` 持有 `v8::Global<v8::Context>`、`SkeletonRegistry`、dispatch function 等强引用
- 旧实现无销毁入口，iframe 生命周期无法闭环
- 长期运行会累积内存，最终 OOM

### P1：GC 弱回调中直接 `erase()`（未定义行为）

- 弱回调 first-pass 直接改 STL 容器，有迭代器失效 / 重入风险
- 可能造成概率性 crash 或静默数据损坏

### P2：定时器嵌套层级为实例级状态（多 Context 串扰）

- 多 iframe 子上下文共享嵌套层级状态
- 可能导致 4ms clamp 指纹/行为错误

### P3：Inspector WebSocket 空闲超时（调试体验问题）

- `idleTimeout=960s` 导致长时间纯断点无数据时断流
- 影响调试体验，但不影响生产逻辑

## 修复（落地实现摘要）

### 1. `child_frames_`：vector → map + 显式销毁 API

- `child_frames_` 改为 `std::unordered_map<int, ChildFrame>`
- 增加单调递增 `frameId`
- 新增 `NativeDestroyChildFrame` / `DestroyChildFrameOnVmThread`
- 注册 `__destroyChildFrame__(frameId)` 全局入口
- 销毁时先 `Reset()` 所有 `v8::Global`，再移除 map 条目

收益：

- 子帧生命周期可闭环，长期运行泄漏风险显著下降

### 2. DOM wrapper 弱回调改延迟清理

- 新增 `pending_dom_wrapper_cleanup_` 队列
- 弱回调阶段只入队，不直接 `erase`
- 在 `RunLoopOnce` 末尾统一清理

收益：

- 避免 GC 回调阶段容器修改带来的未定义行为

### 3. 定时器嵌套层级改为 per-Context（EmbedderData）

- 移除实例级嵌套状态成员
- 使用 V8 Context EmbedderData 存储 `nestingLevel`
- TimerTask 记录 `owner_ctx`
- 执行/清理时恢复对应 Context 状态并释放句柄

收益：

- 多 iframe / 多 context 场景下嵌套层级不再串扰

### 4. Inspector 空闲超时处理策略调整（含回退）

- 历史方案：心跳线程定期 ping（后续回退）
- 当前保留策略：
  - `idleTimeout=960`
  - `sendPingsAutomatically=true`
  - 保留 WebSocket 发送持锁修复
  - 保留 Inspector 协议消息重入排队修复
  - 调整 ready 等待默认值（启动更快）

说明：

- 以交互稳定性优先，接受“长时间空闲断点会断流”的调试体验折中。

### 5. 附加修复：Event skeleton 未进入 bundle
