# 运行时核心与配置

> 源文件：`leap-env/runner.js`，`leap-env/src/core/config.js`，`leap-env/src/core/runtime.js`，`leap-env/src/core/skeleton-loader.js`，`leap-env/src/core/tools.js`，`leap-env/src/instance/host-log.js`，`leap-vm/index.js`
> 更新：2026-03-05

## 功能概述

运行时核心层是 LeapEnv 的控制平面，负责：

1. **环境生命周期**：通过 `runner.js` 导出的三个主函数管理 VM 的初始化、任务执行与销毁。
2. **配置注入**：`runner.js` 在 prelude 阶段写入 `leapenv.__runtimeBootstrap`，`config.js/runtime.js` 读取并归一化配置。
3. **JS 运行时基础**：`runtime.js` 建立 `leapenv` 命名空间、dispatch 函数、descriptor 缓存和路由校验。
4. **Skeleton 加载**：`skeleton-loader.js` 对 skeleton 对象按 profile 过滤后，一次性调用 `leapenv.getNativeBridge().defineEnvironmentSkeleton(...)`。
5. **工具函数**：`tools.js` 提供标签名→构造函数名映射（`getConstructorName`）。
6. **宿主日志**：`host-log.js` 提供带级别过滤的 Node.js 侧日志输出（`hostLog`）。

## 关键机制

### 三主函数（runner.js 公开 API）

| 函数 | 职责 |
|------|------|
| `initializeEnvironment(options)` | 加载 leap-vm、配置 Hook 黑名单、运行 prelude/bundle，返回 `{leapvm, resolved, inspectorInfo}` |
| `executeSignatureTask(leapvm, task)` | 将 begin-scope + snapshot 注入 + targetScript + end-scope 合并为**单次** `runScript` 调用 |
| `shutdownEnvironment(leapvm)` | 释放所有 DOM scope，调用 `leapvm.shutdown()` |

便捷函数 `runEnvironment(options)` = `initializeEnvironment` → `executeSignatureTask` → `shutdownEnvironment`，用于单次运行场景。

### initializeEnvironment 详细流程

```
resolveRunOptions(options)
  └─ normalizeDomBackend()       → 解析 LEAP_DOM_BACKEND（spec→dod 兼容映射）
  └─ normalizeSignatureProfile() → 解析 LEAP_SIGNATURE_PROFILE（默认 fp-lean）
loadLeapVm()                     → 优先 LEAP_VM_PACKAGE_PATH，次选 ../leap-vm，最后 'leap-vm'
configureHooks(leapvm, options)  → setPropertyBlacklist + 可选 setMonitorEnabled + installBuiltinWrappers
loadBundle()                     → 读取 dist/leap.bundle.js（或 options.bundleCode 预加载）
maybeEnableInspector()           → debug 模式下调用 leapvm.enableInspector()
runDebugPrelude()                → 注入 console/leapenv + __runtimeBootstrap（domBackend/signatureProfile/hostTimers/strict hardening）
runBeforeScript()                → 若 options.beforeRunScript 非空，执行用户自定义前置脚本
runEnvironmentBundle()           → try{bundle}catch{throw} 执行 bundle
```

### loadLeapVm 的 Linux worker 加载策略（2026-03-05）

`runner.js` 的 `loadLeapVm()` 最终会进入 `leap-vm/index.js`。在 Linux `worker_threads` 场景下，addon 默认通过：

- `process.dlopen(binaryPath, RTLD_NOW | RTLD_NODELETE)`

加载 `leapvm.node`，避免 worker 退出阶段 `dlclose + TLS` 清理触发原生崩溃。  
该策略默认开启，无需额外配置。

### executeSignatureTask 单次合并执行

为降低每任务 VM 调用开销（从 3-5 次 `runScript` 降为 1 次），将以下步骤内联为单个脚本：

```
setup 阶段（hook inactive）：
  __LEAP_HOOK_RUNTIME__.phase = 'setup'
  beginTaskScope(taskId)
  applyFingerprintSnapshot(...)
  applyStorageSnapshot(..., policy)
  applyDocumentSnapshot(...)
task 阶段（hook active）：
  __LEAP_HOOK_RUNTIME__.phase = 'task' / active = true
  [beforeScript]
  [targetScript]
finally（始终执行）：
  hook active = false / phase = 'idle'
  leapenv.resetSignatureTaskState()
  endTaskScope(taskId)
```

支持通过 `task.resourceName` 传 ScriptOrigin，使 `Error.stack` 显示真实文件名（A1 特性）。

### config.js / runtime.js 配置读取优先级

bundle 内运行，依次读取：

| 配置项 | 优先级 | 可选值 |
|--------|--------|--------|
| `domBackend` | `leapenv.__runtimeBootstrap.domBackend` > `LEAP_DOM_BACKEND` > 默认 `dod` | `dod`（唯一，默认）；遗留值 `spec` 映射到 `dod` |
| `signatureProfile` | `leapenv.__runtimeBootstrap.signatureProfile` > `LEAP_SIGNATURE_PROFILE` > 默认 `fp-lean` | `fp-lean`（默认）/ `fp-occupy` |
| `bridgeExposureMode` | `leapenv.__runtimeBootstrap.bridgeExposureMode` > 运行时默认 | **固定 `strict`**（默认不暴露 `$native/__createNative__` 等全局名） |
| `globalFacadeMode` | `leapenv.__runtimeBootstrap.globalFacadeMode` > 运行时默认 | **固定 `strict`**（默认最小 facade + 冻结 `globalThis.leapenv`） |
| `dispatchMissingMode` | 硬编码 | `warn`（默认，控制台警告）/ `silent`（静默返回 undefined）/ `throw`（抛出 TypeError） |
| `validateWarn` | 硬编码 | `false`（默认）/ `true`（逐条路由警告） |
| `envVersion` | `LEAP_ENV_VERSION` > 硬编码 `1.0.0` | semver 字符串 |

注意：当前默认路径不再依赖 `applyDomBackendSetting` / `applySignatureProfileSetting`，而是由 `runDebugPrelude` 一次性写入 `leapenv.__runtimeBootstrap`。`runtime.js` 仍保留对旧 `__LEAP_*` 全局 key 的兼容读取与清理。

### 并发池传输模式补充（非 runner 配置链路）

以下配置不属于 `runner.js -> config.js` 的 bundle 配置解析链路，而是由并发池模块（`leap-env/src/pool/thread-pool.js`）在 Node.js 宿主侧读取：

| 配置项 | 读取位置 | 默认值 | 可选值 |
|--------|----------|--------|--------|
| `dodTransferMode` | `new ThreadPool({ dodTransferMode })` | `transfer` | `transfer` / `clone` |
| `LEAP_DOD_TRANSFER_MODE` | `process.env.LEAP_DOD_TRANSFER_MODE` | `transfer` | `transfer` / `clone` |

说明：
- `transfer`：DoD tree 通过 transferables 零拷贝传输（默认）。
- `clone`：回退为结构化拷贝模式（兼容兜底）。
- 该开关只影响线程池任务分发阶段，不影响 bundle 内 `config.js` 的 `domBackend/signatureProfile` 解析。

### runtime.js 核心职责

- **`leapenv` 命名空间初始化**：保证幂等（`_runtimeInitialized` 守卫），建立 `config / toolsFunc / impl / implRegistry / memory` 各命名空间。
- **`leapenv.registerImpl(typeName, implClass)`**：注册 impl 类并预缓存整条原型链的 `getOwnPropertyDescriptor`（O1 优化），避免 dispatch 热路径重复调用。
- **`runtimeStore.bridge.dispatch = dispatch`**：统一分发函数入口，C++ stub 通过 `leapenv.__runtime.bridge.dispatch` 调用 JS impl。支持 GET / SET / CALL 三种 actionType，优先使用 O1 descriptor 缓存。
- **`leapenv.validateDispatchRoutes()`**：全量校验 skeleton 与 impl 注册的路由一致性，`validateWarn=true` 时逐条输出，否则仅汇总计数。
- **`Error.prepareStackTrace` 重写**：过滤 leap-env 内部帧，格式化为 Chrome 风格堆栈（A2 特性），带再入守卫防递归。
- **`LEAP_TRACE=1` 追踪日志**：每次 dispatch 调用时打印 `{t, type, prop, action}` JSON（I-8 特性）。
- **全局收敛（默认 strict）**：启动后将 native bridge 注入内部 runtime bridge，并移除 `$native`/`__createNative__`/`__createChildFrame__` 等全局桥接名；同时将 `globalThis.leapenv` 锁定为最小可见 facade。

### skeleton-loader.js 加载流程

```
leapenv.loadSkeleton()
  filterSkeletonObjectsForProfile(leapenv.skeletonObjects)
    └─ fp.resolveProfile(profileName) → objectPolicy + windowInstance policy
    └─ 按对象级 hide/allow/placeholder 过滤 skeleton 列表
    └─ window.instance 单独过滤属性级 hide/allow/placeholder
  leapenv.getNativeBridge().defineEnvironmentSkeleton({ schemaVersion:1, envVersion, objects })
```

过滤统计写入 `leapenv.lastSkeletonProfileStats`（含 `hiddenObjects / hiddenWindowProps / placeholderWindowProps / allowlistedWindowProps`）。

### tools.js

`leapenv.toolsFunc.getConstructorName(tagName)`：将 HTML 标签名映射到对应 WebIDL 构造函数名，仅覆盖已明确映射的标签（div→HTMLDivElement、span→HTMLSpanElement、a→HTMLAnchorElement、canvas→HTMLCanvasElement、iframe→HTMLIFrameElement），未收录标签返回 `HTMLUnknownElement`。

### host-log.js

Node.js 进程侧日志工具，通过 `LEAPVM_HOST_LOG_LEVEL` 环境变量控制最低输出级别（`debug/info/warn/error`，默认 `info`）。仅用于 runner.js 内部的宿主侧日志，不进入 VM 沙箱。

## 主要流程

```
外部调用方
    │
    ├─ initializeEnvironment(options)
    │      │
    │      ├─ loadLeapVm()  ──────────────────────────→ VmInstance (C++ addon)
    │      ├─ configureHooks()  ─────────────────────→ setPropertyBlacklist / setMonitorEnabled
    │      ├─ loadBundle()  ──────────────────────────→ dist/leap.bundle.js
    │      ├─ maybeEnableInspector()  ───────────────→ leapvm.enableInspector()
    │      ├─ runDebugPrelude()  ────────────────────→ VM: 全局初始化脚本
    │      ├─ runBeforeScript()  ────────────────────→ VM: 用户自定义前置脚本（可选）
    │      └─ runEnvironmentBundle()  ───────────────→ VM: bundle（含 runtime→config→skeleton→impl）
    │
    ├─ executeSignatureTask(leapvm, task)  ──────────→ VM: 单次合并脚本执行
    │
    └─ shutdownEnvironment(leapvm)  ─────────────────→ VM: releaseAllScopes + leapvm.shutdown()
```

Bundle 内部加载顺序（由构建系统保证）：

```
runtime.js              → leapenv 命名空间 + runtime bridge dispatch
config.js               → 配置读取
fingerprint-profile.js  → fp-lean / fp-occupy profile 定义
tools.js                → getConstructorName
placeholder-policy.js   → placeholder 错误工厂
skeleton-loader.js      → filterSkeletonObjectsForProfile + loadSkeleton
*.skeleton.js           → leapenv.skeletonObjects 填充
*.impl.js               → leapenv.registerImpl 注册
skeleton-init.instance.js → leapenv.loadSkeleton() 触发（所有 impl 注册完成后）
```
