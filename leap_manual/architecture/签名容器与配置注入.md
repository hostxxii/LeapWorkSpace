# 签名容器与配置注入

> 源文件：`leap-env/src/core/fingerprint-profile.js`，`leap-env/src/core/placeholder-policy.js`，`leap-env/src/core/skeleton-loader.js`，`leap-env/src/instance/signature-task.instance.js`，`leap-env/runner.js`
> 更新：2026-03-06

## 功能概述

本模块承担两项职责：

1. **Profile 暴露面控制**：通过 `fp-lean` / `fp-occupy` 两种 profile 规则，决定签名容器对目标脚本暴露哪些 Web API（hide / placeholder / keep），过滤在 skeleton 传入 C++ 之前完成于 JS 侧。
2. **siteProfile 任务态注入**：`runner.js` 接收每次任务的 `siteProfile`，调用 `applyFingerprintSnapshot` / `applyStorageSnapshot` / `applyDocumentSnapshot` 注入 `signatureTaskState`，impl 读取此状态返回站点定制值，任务结束后 `resetSignatureTaskState()` 清理。
3. **职责边界约束**：`siteProfile` 只承载任务态覆盖数据，不承载编译后的脚本产物、常驻函数句柄或任何执行缓存元数据；这些内容属于 `runner.js` 的宿主侧执行模型。

## 关键机制

### 1. API 分层（Tier A / B / C）

`fingerprint-profile.js` 将 Web API 分为三层，作为 profile 规则的参考依据：

| Tier | 含义 | 典型对象 |
|------|------|----------|
| **A** | 核心 DOM/BOM，签名脚本必须访问 | `Window`、`Document`、`Navigator`、`Location`、`Screen`、`Performance`、`Storage`、`Event` 等 |
| **B** | 扩展 Web API，可能被脚本探测 | `DOMParser`、`XMLSerializer`、`XMLHttpRequest`、`fetch` 等 |
| **C** | 网络/通信 API，高风险 | `WebSocket`、`Worker`、`SharedWorker`、`BroadcastChannel`、`IndexedDB` 等 |

### 2. fp-lean 与 fp-occupy 规则

| Profile | window.instance Tier B | window.instance Tier C |
|---------|------------------------|------------------------|
| `fp-lean`（默认） | **hide**（属性不存在，`typeof === 'undefined'`） | **hide** |
| `fp-occupy` | **placeholder**（属性存在，调用时抛标准错误） | **hide** |

两种 profile 的 `objectPolicy.defaultAction` 均为 `keep`，即 skeleton 对象本身不过滤；过滤粒度仅到 `window.instance` 属性级别。

### 3. filterSkeletonObjectsForProfile 执行流程

```
leapenv.config.signatureProfile
    ↓
fingerprintProfile.resolveProfile(name)
    ↓
filterSkeletonObjectsForProfile(skeletonObjects)
    ├── 每个 skeleton 对象：objectPolicy 决定 keep / hide（当前均 keep）
    └── window.instance：逐属性应用 windowInstance policy → hide / placeholder / keep
    ↓
过滤后 descriptor → $native.defineEnvironmentSkeleton(descriptor)
    ↓
C++ 构建 V8 模板（hide 属性完全不注册，placeholder 属性正常注册，行为由 impl 决定）
```

过滤统计写入 `leapenv.lastSkeletonProfileStats`（含 `hiddenWindowProps`、`placeholderWindowProps` 等字段）。

### 4. placeholderPolicy（7 个标准错误工厂）

`placeholder-policy.js` 提供 impl 在 placeholder 属性被调用时使用的统一接口：

| 方法 | 用途 | 错误码 |
|------|------|--------|
| `networkDisabledError(apiName)` | 网络 API 被禁用，返回 `Error` | `LEAP_NETWORK_DISABLED` |
| `rejectNetwork(apiName)` | 同上，返回 `Promise.reject(...)` | `LEAP_NETWORK_DISABLED` |
| `notImplementedError(apiName)` | API 未实现，返回 `Error` | `LEAP_NOT_SUPPORTED` |
| `invalidStateError(apiName)` | 无效状态，返回 `Error` | `LEAP_INVALID_STATE` |
| `createTypeError(message, code)` | 通用 TypeError | 自定义或 `LEAP_PLACEHOLDER_TYPE_ERROR` |
| `emptyHeaders()` | 返回空 Headers 对象 | — |
| `emptyRecords()` | 返回空记录结构 | — |

### 5. siteProfile 结构与字段

```json
{
  "id": "jd-search-pc",
  "overrideMode": "strict",
  "requiredFields": ["fingerprintSnapshot.navigator.userAgent"],
  "fingerprintSnapshot": {
    "navigator": { "userAgent": "...", "platform": "Win32", "languages": ["zh-CN"] },
    "screen": { "width": 1920, "height": 1080 },
    "location": { "href": "https://search.jd.com/..." }
  },
  "storagePolicy": { "localStorage": "replace", "sessionStorage": "replace" },
  "storageSnapshot": { "localStorage": { "k1": "v1" } },
  "documentSnapshot": { "cookie": "a=1; b=2", "referrer": "https://www.jd.com/" }
}
```

`overrideMode: merge`（默认）只覆盖已提供字段；`overrideMode: strict` 配合 `requiredFields`，缺字段时任务执行前直接失败。

### 5.1 siteProfile 的边界

`siteProfile` 允许承载的内容仅限任务态快照：

- `fingerprintSnapshot`
- `storagePolicy`
- `storageSnapshot`
- `documentSnapshot`

`siteProfile` 不应承载：

- 编译后的 `targetScript` 产物
- code cache buffer
- 任意常驻脚本/函数句柄
- 与单个 `VmInstance` 生命周期绑定的执行缓存键

原因是这些内容不是“站点配置覆盖”，而是 runner 在宿主侧对任务执行方式的优化实现。将编译产物并入配置 JSON，会把“任务态数据”和“执行态缓存”混为一层，既增加传输成本，也会破坏配置注入链路的职责边界。

### 6. signatureTaskState 对象

`signature-task.instance.js` 维护 `leapenv.signatureTaskState`，由以下三个注入函数填充：

| 注入函数 | 填充字段 |
|----------|----------|
| `applyFingerprintSnapshot(snap)` | `navigator`、`screen`、`windowMetrics`、`performanceSeed`、`featureFlags`、`randomSeed`、`canvasProfile` |
| `applyStorageSnapshot(snap, policy)` | `localStorage`、`sessionStorage`（按 `replace` / `merge` 策略） |
| `applyDocumentSnapshot(snap)` | `cookie`、`referrer`、`lastModified` |

任务结束后 `resetSignatureTaskState()` 将所有字段清空。

## 主要流程

```
环境初始化：
  leapvm.runScript(bundleCode)
    → IIFE 执行，skeleton-loader.js 调用 filterSkeletonObjectsForProfile()
      → 按 signatureProfile（fp-lean/fp-occupy）过滤 skeletonObjects
        → $native.defineEnvironmentSkeleton(filteredDescriptor)
          → C++ 构建 V8 模板（hide 属性不注册，placeholder 属性正常注册）

任务执行前（注入）：
  runner.js: executeSignatureTask(leapvm, task)
    → applyFingerprintSnapshot(task.siteProfile.fingerprintSnapshot)
    → applyStorageSnapshot(task.siteProfile.storageSnapshot, storagePolicy)
    → applyDocumentSnapshot(task.siteProfile.documentSnapshot)
      → leapenv.signatureTaskState 填充完毕

任务执行时（宿主侧优化，独立于 siteProfile）：
  runner.js
    → 小脚本：单次合并 runScript
    → 大脚本：setup 小脚本 + 稳定目标源码 + 每 VmInstance code cache + cleanup 小脚本
      → code cache 保存于 leapvm.__leapTaskExecutionCache
      → 与 siteProfile JSON 无耦合

任务执行中：
  目标脚本访问 navigator.userAgent
    → NavigatorImpl.userAgent getter 读取 leapenv.signatureTaskState.navigator.userAgent
      → 返回注入值（strict 模式下缺字段已提前失败）

任务结束后（清理）：
  runner.js: resetSignatureTaskState()
    → leapenv.signatureTaskState 全部清空，准备下一任务
```
