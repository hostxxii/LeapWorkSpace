# 构建与打包系统

> 源文件：`leap-env/src/build/build.js`，`leap-env/src/build/entry.js`，`leap-env/src/build/generate-entry.js`，`leap-env/src/build/validate-skeleton-context.js`
> 更新：2026-03-01

## 功能概述

构建系统将 `leap-env/src/` 下所有 JS 模块打包成单个 `leap.bundle.js`（IIFE 格式），供 `runner.js` 在 VM 内执行。整个流程由 esbuild 驱动，入口文件 `entry.js` 由 `generate-entry.js` 自动扫描生成，确保模块加载顺序正确。

## 关键机制

### 1. 构建三步流程

```
npm run build  →  build.js
  ① require('./generate-entry.js')          // 扫描 src/ 生成 entry.js
  ② validateSkeletonContext(src_dir)         // 校验 skeleton 文件执行上下文
  ③ esbuild.build(buildOptions)             // 打包 → dist/leap.bundle.js
```

### 2. generate-entry.js：自动扫描与排序

扫描 `leap-env/src/`（排除 `build/`、`pool/`、`entry.js` 自身），将所有 `.js` 文件按所属目录分入 7 个分组：

| 分组 | 目录 | 排序规则 |
|------|------|----------|
| `core` | `src/core/` | 按 `CORE_ORDER` 显式顺序（见下） |
| `skeleton-type` | `src/skeleton/type/` | 继承拓扑排序（父类优先） |
| `skeleton-instance` | `src/skeleton/instance/` | `window.instance.skeleton.js` 强制最前，其余字母序 |
| `skeleton-other` | `src/skeleton/`（其余） | 字母序 |
| `impl` | `src/impl/` | 字母序 |
| `instance` | `src/instance/` | 字母序 |
| `other` | 其余 | 字母序 |

**CORE_ORDER**（`runtime.js` → `config.js` → `fingerprint-profile.js` → `tools.js` → `placeholder-policy.js` → `skeleton-loader.js`）确保核心模块按依赖顺序加载。

**skeleton/type 拓扑排序**：读取每个 skeleton 文件的 `"super"` 字段，建立父子关系图，用 Kahn 算法（BFS 入度）生成父类优先的加载顺序。循环依赖检测：若拓扑排序后节点数不足，发出 Warning 并追加剩余节点（字母序兜底）。

### 3. validate-skeleton-context.js

扫描 `src/skeleton/` 下所有 `.js` 文件，检查是否存在 `})(this);` 字符串。Skeleton 文件必须使用 `})(globalThis);`，否则在 IIFE 环境中 `this` 会绑定到 `exports`，导致 `leapenv` 写入错误对象。发现违规则构建失败退出。

### 4. esbuild 配置

| 配置项 | 值 | 说明 |
|-------|----|------|
| `entryPoints` | `src/build/entry.js` | 自动生成的入口 |
| `bundle` | `true` | 合并所有 `import` |
| `outfile` | `src/build/dist/leap.bundle.js` | 产物路径 |
| `format` | `iife` | 立即执行函数包装，不依赖模块系统 |
| `platform` | `neutral` | 不注入 Node.js 或浏览器特有 polyfill |
| `target` | `es2020` | 语法降级目标 |
| `minify` | `false` | 保留可读性，便于调试 |
| `sourcemap` | `true` | 生成 `leap.bundle.js.map` |

### 5. Watch 模式

`node build.js --watch` 进入增量构建模式（`esbuild.context(opts).watch()`），文件修改时自动重新打包。`generate-entry.js` 在启动时执行一次扫描；Watch 过程中不重新扫描（新增模块文件需手动重启）。

### 6. pool/ 排除

`generate-entry.js` 在扫描时显式跳过 `pool/` 目录。池化模块（`thread-pool.js`、`process-pool.js`、`worker.js` 等）是 Node.js 宿主代码，运行在主机进程，不应进入 VM 内的 bundle。

## 主要流程

```
开发工作流：
  修改 src/ 任意模块
    → npm run build
      → generate-entry.js 重新扫描，更新 entry.js
      → validateSkeletonContext 检查
      → esbuild 打包 → dist/leap.bundle.js

Runner 加载：
  runner.js → loadBundle(bundlePath)
    → 读取 dist/leap.bundle.js（或 bundleCode 参数跳过磁盘读取）
      → leapvm.runScript(bundleCode)
        → IIFE 在 VM 上下文内自执行，初始化 leapenv

工作线程预读：
  ThreadPool worker 启动时读取 bundlePath → bundleCode 缓存到内存
    → 后续任务通过 bundleCode 参数传入，跳过磁盘 IO
```
