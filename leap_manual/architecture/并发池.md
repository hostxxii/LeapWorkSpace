# 并发池

> 源文件：`leap-env/src/pool/thread-pool.js`，`leap-env/src/pool/process-pool.js`，`leap-env/src/pool/worker.js`，`leap-env/src/pool/worker-common.js`，`leap-env/src/pool/thread-worker.js`
> 更新：2026-03-03

## 功能概述

并发池为外部调用方提供签名任务的并发执行能力，支持两种后端：

- **线程池（ThreadPool）**：基于 `worker_threads`，worker 与主进程同 PID，共享内存空间，支持 ArrayBuffer 零拷贝传输（DoD 优化路径），**生产主线**
- **进程池（ProcessPool）**：基于 `child_process.fork`，每个 worker 独立进程，隔离更强，**稳定回退**

两者对外暴露相同接口（`start` / `runSignature` / `close` / `getStats`），调用方可无缝切换。

> **注意**：并发池**不是** runner 内部模块，没有通过 `runner.js` 导出，也没有运行时选择逻辑（无 `LEAP_POOL_MODE` 注入入口）。调用方（测试/应用脚本）直接 `require` 对应池文件并自行实例化。

## 关键机制

### 1. Worker 生命周期（两池相同）

```
构造 → start() → _spawnWorker() × N → 并行等待 init ack
                                          ↓
                                  worker 收到 {type:'init'} 消息
                                  → initializeEnvironment() → leapvm 就绪
                                  → 回发 {type:'init', warmupMs}
                                          ↓
                               pool 将 workerId 加入 idleWorkerIds
                               → _drainQueue() 驱动待处理任务
```

### 2. 任务调度流程

```
runSignature(payload) → pendingQueue.push → _drainQueue()
                                               ↓
                                  idleWorkerIds 取出一个 worker
                                  _assignTask() → postMessage {type:'run_signature'}
                                               ↓
                                  worker: executeSignatureTask() → runPostTaskCleanup()
                                  → 回发 {type:'result'|'error', shouldRecycle?}
                                               ↓
                                  pool: resolve/reject → _releaseWorkerAfterTask()
                                  → 若达到 maxTasksPerWorker → _recycleWorker()
```

### 3. 线程池与进程池的关键差异

| 特性 | ThreadPool | ProcessPool |
|------|-----------|------------|
| 通信方式 | `worker.postMessage` / `parentPort` | `child.send` / `process.send` |
| 零拷贝传输 | 支持（ArrayBuffer transferables） | 不支持（IPC 序列化） |
| DoD 传输默认策略 | `transfer`（零拷贝默认） | `clone`（IPC 序列化） |
| 崩溃隔离 | 同进程，崩溃波及主进程 | 独立进程，完全隔离 |
| 关闭时 VM teardown | **跳过**（Windows SIGSEGV 已知问题） | 正常调用 `shutdownEnvironment()` |
| 启动 bundle 预读 | 支持（pool 层预读一次，workers 跳过磁盘 IO） | 不支持预读（每个 child fork 自行加载） |
| DoD 引擎 | 初始化时按 `domBackend` 动态加载 | 不涉及 DoD 路径 |

### 4. worker-common.js 共享工具

`worker.js`（进程 worker）、`thread-worker.js`（线程 worker）和 `thread-pool.js`（池控制器）均导入 `worker-common.js`，消除重复实现：
- `toPositiveInteger(value, fallback)`：安全整数解析（带正数校验和回退值）
- `getMemorySnapshot()`：进程内存快照
- `serializeError()`：Error → 可传输对象
- `buildPostTaskCleanupScript()`：构造 DOM 泄漏检测脚本
- `CLEANUP_FAILURE_LIMIT`：读取 `LEAP_WORKER_CLEANUP_FAIL_LIMIT`，默认 3

### 5. 心跳与健康监控

- Worker 每隔 `heartbeatIntervalMs`（默认 5000ms）向 pool 回报 `{type:'heartbeat', tasksHandled, memoryUsage}`
- Pool 侧定时器检查：若距上次心跳超过 `heartbeatTimeoutMs`（默认 heartbeatIntervalMs × 3）则强制杀死并重生 worker
- `tasksHandled >= maxTasksPerWorker`（默认 200）时触发主动 recycle

### 6. 清理失败计数（cleanupFailureCount）

任务完成后，worker 在 VM 内运行后清理脚本：
- 调用 `__LEAP_DOM__.releaseTaskScope(taskId)` 检测泄漏
- 每次泄漏文档或清理异常累加 `cleanupFailureCount`
- 达到 `CLEANUP_FAILURE_LIMIT` 后，worker 在回报 result/error 时附带 `shouldRecycle: true`，pool 收到后主动 recycle 该 worker

### 7. DoD 零拷贝（线程池专属）

`_assignTask()` 在分发前扫描 `payload.dodTree`，将所有 TypedArray 的 `.buffer` 收集为 transferables 列表，通过 `postMessage(msg, transferables)` 实现零拷贝传输，消除序列化开销。

当前策略：
- 默认 `transfer`（零拷贝开启）
- 显式回退 `clone`：
  - 构造参数：`new ThreadPool({ dodTransferMode: 'clone' })`
  - 环境变量：`LEAP_DOD_TRANSFER_MODE=clone`

实现位置：`ThreadPool` 构造函数中的 `dodTransferMode` / `enableDodZeroCopy` 推导逻辑。

### 8. 多 isolate 稳定性约束（Dispatch 侧）

`worker_threads` 场景下，每个 worker 都有独立 V8 isolate。Dispatch 品牌校验使用的 `[[leapvm_brand]]` key 必须按 isolate 获取，不能使用进程级静态 V8 句柄缓存（例如 `static v8::Eternal<...>` 跨 isolate 复用），否则可能触发原生访问冲突（`0xC0000005`）。

当前实现已按 isolate 获取品牌 key（见 `leap-vm/src/leapvm/skeleton/dispatch_bridge.cc`）。

## 主要流程

### 启动（线程池）

```
ThreadPool.start()
  → 主进程读取 bundlePath 到内存（bundleCode）
  → 并发 _spawnWorker() × size
      → new Worker(thread-worker.js)
      → postMessage {type:'init', runnerOptions, heartbeatIntervalMs}
      → thread-worker: initializeEnvironment() → leapvm 就绪
      → if domBackend==='dod': require dod-layout-engine, 注册到 globalThis
      → postMessage {type:'init', warmupMs, inspectorInfo}
  → _startHeartbeatMonitor()
```

### 关闭（线程池）

```
ThreadPool.close()
  → closing = true
  → _rejectPendingQueue()     // 队列中等待的任务全部 reject
  → 对每个 worker: postMessage {type:'shutdown'} + forceKill 计时器
  → thread-worker handleShutdown(): 发最后一次 heartbeat → process.exit(0)
  → （注意：thread-worker 不调用 shutdownEnvironment，跳过 VM teardown）
  → 超时后 _forceKillWorker → worker.terminate()
```

### 进程池差异

进程池的关闭在 worker.js 的 `handleShutdown()` 中**正常调用** `shutdownEnvironment(leapvm)`，然后 `process.exit(0)`。
